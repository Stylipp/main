  <name>Task 1: Create SwipeCardStack with AnimatePresence</name>
  <files>apps/stylipp.com/src/features/feed/components/SwipeCardStack.tsx</files>
  <action>
Build SwipeCardStack as the deck manager that orchestrates card rendering, exit animations, and swipe callbacks.

**Component structure:**
- Read cards and currentIndex from useSwipeStore
- Calculate visibleCards = cards.slice(currentIndex, currentIndex + 3) — only render top 3
- Track exitX with useState (set before advancing card)
- Wrap visibleCards in AnimatePresence (NOT mode="wait" — we want overlap during exit)

**Card rendering:**
For each visible card at index i (0 = top, 1 = middle, 2 = bottom):
- key={card.product_id} — required for AnimatePresence to track
- isTop={i === 0} — only top card is draggable
- Stack visual via animate prop:
  - scale: 1 - i * 0.05 (top=1.0, middle=0.95, bottom=0.90)
  - y: i * 8 (top=0, middle=8px, bottom=16px behind)
  - zIndex: 3 - i
- initial={{ scale: 0.9, y: 16 }} for cards entering the stack from below
- transition={{ type: 'spring', stiffness: 300, damping: 25 }}

**Swipe handler:**
When SwipeCard calls onSwipe(direction):
1. Set exitX based on direction: right → 300, left → -300
2. Determine action: right → 'like', left → 'dislike'
3. Call swipeStore.advanceCard(action) — increments index, pushes to undo stack
4. Call onFeedback(currentCard.product_id, action) — passed from parent or from useFeedbackSubmit

**Container styling:**
- position: relative, centered in viewport
- Fixed height (e.g., 480px mobile, 560px desktop) to prevent layout shift
- overflow: visible so exit animations aren't clipped

**Important:**
- Do NOT use AnimatePresence mode="wait" — it delays entry of next card until exit completes. Use default mode (simultaneous) for smooth stack transitions.
- The exitX state must be set BEFORE advanceCard() to ensure the exiting card reads the correct exit direction.

<task type="auto">
  <name>Task 2: Create useFeedbackSubmit hook with optimistic submission</name>
  <files>apps/stylipp.com/src/features/feed/hooks/useFeedbackSubmit.ts</files>
  <action>
Build useFeedbackSubmit hook for fire-and-forget feedback submission with retry.

**Hook interface:**
- Returns: { submitFeedback: (productId: string, action: SwipeAction) => void, retryPending: () => void }

**submitFeedback logic:**
1. Call feedbackService.submitFeedback(productId, action) — do NOT await in the UI flow
2. Use .catch() to handle failures:
   - On failure: add to swipeStore.addPendingFeedback({ productId, action, retryCount: 0 })
   - Log warning but do NOT show error to user — swipe must feel instant
3. On success: if the item was in pendingFeedback queue, remove it

**retryPending logic:**
- Iterate over pendingFeedback from swipeStore
- For each, re-attempt submitFeedback
- If retryCount >= 3, remove from queue (give up silently — data is not critical enough to block UX)
- Increment retryCount on each retry attempt

**Undo support:**
- Export undoLastSwipe function that:
  1. Calls swipeStore.undoLastSwipe() to get the SwipeRecord
  2. If record exists, submit a corrective feedback: the opposite action or a special 'undo' signal
  3. For MVP: just send the new action — backend allows multiple interactions per user+product, so the latest record wins for learning purposes

**Auto-retry:**
- Use useEffect with a 30-second interval to call retryPending() if pendingFeedback.length > 0
- Clean up interval on unmount

Do NOT block the UI on any API call. The principle: swipe feels instant, backend syncs in background.
