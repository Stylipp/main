  <action>
    **Schemas (schemas.py):**

    Create Pydantic v2 models:

    - `FeedItem(BaseModel)`: product_id (str), title (str), price (float), currency (str), image_url (str), product_url (str), score (float), explanation (str — one of 3 templates per PROJECT.md: "Similar to your recent likes", "Matches your style", "Within your usual price range"). Use `model_config = ConfigDict(from_attributes=True)`.

    - `FeedResponse(BaseModel)`: items (list[FeedItem]), next_cursor (str | None), has_more (bool), total_in_batch (int).

    **Explanation template selection logic:** Simple rule — if price_score is the highest non-cosine factor → "Within your usual price range", if cluster_prior_score is highest → "Matches your style", default → "Similar to your recent likes". This is the template-only explainability from PROJECT.md (no category labeling).

    **Router (router.py):**

    Create FastAPI APIRouter with prefix="/feed", tags=["feed"].

    `GET /` (becomes GET /api/feed via main.py prefix):
    - Query params: `cursor: str | None = None`, `page_size: int = Query(default=20, ge=1, le=50)`
    - Dependencies: `user_id = Depends(get_current_user)`, `session = Depends(get_async_session)`
    - Logic:
      1. If cursor provided: decode base64 JSON to get offset and batch_id (validate, return 400 on invalid cursor)
      2. Instantiate FeedService with Qdrant client from `get_qdrant_client()` and settings
      3. Call `feed_service.generate_feed(user_id=str(user_id), seen_ids=[], session=session, page_size=page_size)`
         - For cursor pagination: seen_ids starts empty (initial load), frontend tracks seen items client-side for now. The offset in cursor skips already-returned items from the ranked batch.
      4. Enrich ranked candidates with product metadata from PostgreSQL (title, image_url, product_url, currency) — query Product model by product_ids
      5. Build FeedItem list with explanation templates
      6. Encode next_cursor if more items available
      7. Return FeedResponse

    **Cursor encoding/decoding:** Use base64.urlsafe_b64encode/decode with JSON payload {"o": offset, "b": batch_id} per RESEARCH.md code example. batch_id can be a simple timestamp or UUID for cache key (without Redis, batch_id is informational only for now).

    **What to avoid and WHY:**
    - Do NOT add Redis caching for feed batches — deferred to Phase 12 per RESEARCH.md Pitfall 5
    - Do NOT add complex seen-item tracking server-side — frontend tracks seen IDs for MVP, server-side seen tracking is Phase 12 optimization
    - Do NOT create more than 3 explanation templates — PROJECT.md explicitly limits to 3 templates only
    - Do NOT use offset-based pagination with page numbers — cursor-based is more stable for dynamic feeds per RESEARCH.md

    Create empty __init__.py files for schemas/ and router/ packages.
