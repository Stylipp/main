<task type="auto">
  <name>Task 1: Create FeedService with candidate retrieval and multi-factor ranking</name>
  <files>apps/backend/src/features/feed/service/feed_service.py</files>
  <action>
    Create `FeedService` class with constructor accepting `qdrant_client: AsyncQdrantClient` and `settings: Settings`.

    Implement `generate_feed(user_id: str, seen_ids: list[str], session: AsyncSession, page_size: int = 20) -> list[RankedCandidate]`:

    1. **Load user profile:** Query PostgreSQL `users` table for `price_profile` JSONB. Query Qdrant `user_profiles` collection for user vector by point ID = user_id (same pattern as onboarding stores it).

    2. **Retrieve candidates from Qdrant:** Use `qdrant_client.search()` (NOT `query_points()` — Qdrant v1.7.4 does not support it).
       - collection_name="products"
       - query_vector=user_vector (768-dim)
       - limit=100 (overretrieve for re-ranking headroom)
       - score_threshold=0.2 (conservative floor per RESEARCH.md)
       - with_payload=True, with_vectors=False
       - query_filter: must_not HasIdCondition for seen_ids, must FieldCondition price range (price_min * 0.5 to price_max * 2.0) — generous per RESEARCH.md to avoid over-filtering

    3. **Apply ranking:** Call `rank_candidates()` from Plan 01's ranking_service with candidates, user_price_profile, and cluster_priors (load from Qdrant style_clusters collection — same pattern as cold_start_service).

    4. **Return top page_size items** from ranked list (diversity injection is Task 2).

    **Shortfall handling (Pattern 4 from RESEARCH.md):**
    - If < page_size candidates after filtering: retry without price filter
    - If still < page_size: retry without seen_ids exclusion (allow revisits)
    - Log shortfall occurrences for monitoring

    **What to avoid and WHY:**
    - Do NOT use `qdrant_client.query_points()` — it maps to a newer API endpoint unsupported by Qdrant v1.7.4 server (RESEARCH.md Pitfall 6)
    - Do NOT cache cluster priors in this plan — caching is explicitly deferred to Phase 12 (RESEARCH.md Pitfall 5)
    - Do NOT duplicate the cold_start_service's Qdrant compatibility workarounds — use the standard `search()` method directly, which works on v1.7.4

    **Helper: `_load_cluster_priors()`** — Scroll all points from `style_clusters` collection, build dict mapping cluster_id → prior_probability from payload. This is a small collection (~10-200 clusters), so full scan is fine.
  </action>

  <action>
    Add diversity injection to `generate_feed()`, following the established pattern from `cold_start_service._interleave_diversity()`.

    **Diversity retrieval:**
    1. After loading cluster priors in Task 1, identify user's primary clusters (top 3 by prior * cosine affinity) and diversity clusters (next 2 adjacent — same pattern as cold_start_service: 4th-5th ranked clusters).
    2. Retrieve diversity candidates: search Qdrant products collection with user_vector but filtered to diversity cluster IDs only. Limit ~20 candidates.
    3. Rank diversity candidates through same ranking pipeline.

    **Diversity injection (mandatory per PROJECT.md):**
    - Reserve 3 slots out of page_size (20) for diversity items
    - Take top 17 from primary ranking, top 3 from diversity ranking
    - Interleave diversity items evenly across the feed (every ~6th position)

    - Do NOT duplicate cold_start_service code — write a similar but feed-specific implementation since cold_start operates on photo embeddings while feed operates on user vectors. The pattern is the same but the input data differs.
