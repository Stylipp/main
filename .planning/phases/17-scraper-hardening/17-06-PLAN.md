<task type="auto">
  <name>Task 2: Add end-to-end regression tests for the hardened scraper flow</name>
  <files>apps/backend/tests/features/products/test_ingest_batch.py, apps/scraper/tests/test_backend_sync.py, apps/scraper/tests/test_change_detection.py, apps/scraper/tests/test_pipeline.py, apps/scraper/tests/test_product_scraper.py</files>
  <action>
Add focused tests that cover the exact failure modes this phase is meant to eliminate:

1. Backend batch ingest response:
- HTTP 200 full success returns `accepted_ids`
- HTTP 207 partial success returns both `accepted_ids` and `rejected`

2. Scraper sync and retry:
- `BackendSync.push_products()` and `update_products()` parse `accepted_ids`
- pipeline updates hashes only for accepted products
- rejected products keep old/no hash and therefore reappear on the next run

3. Removal and return flow:
- `mark_removed()` soft-deletes instead of hard-deleting
- removed products trigger backend archive notification
- a returning product clears `removed_at` and is treated as returning, not duplicated

4. Stable ID extraction:
- JSON-LD `sku` is preferred
- WooCommerce `data-product_id` is used when `sku` is absent
- fallback stays on `md5_...` when no stable source id exists

These tests should be fast unit/integration tests around the affected components. Do not rely on a live backend or live Qdrant.
  </action>
  <verify>cd apps/backend && python -m pytest -p no:cacheprovider tests/features/products/test_ingest_batch.py && cd ../.. && cd apps/scraper && python -m pytest tests</verify>
  <done>Phase 17 has regression coverage for batch result parsing, selective hash updates, removal/relist flows, and stable ID extraction.</done>
</task>
