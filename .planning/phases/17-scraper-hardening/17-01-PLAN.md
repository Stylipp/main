<objective>
Update the batch ingest API to return per-product success/failure IDs instead of aggregate counts.

Purpose: The scraper currently can't distinguish which products succeeded or failed during batch ingestion. Without per-item IDs, hash updates are applied blindly — failed products never get retried because their hashes are already recorded in SQLite.
Output: Updated `BatchIngestResponse` schema with `accepted_ids` and `rejected` arrays, router returning HTTP 207 on partial success.
</objective>

  <action>
Modify the `ingest_batch` endpoint:

1. Replace the `stats["errors"]` list with two new trackers: `accepted_ids: list[str] = []` and `rejected: list[RejectedItem] = []`.

2. In the success branch (`result.success`), append `product.external_id` to `accepted_ids`.

3. In the failure branches (both caught exceptions and `not result.success`), append a `RejectedItem` to `rejected`. Set `retryable=True` for unexpected exceptions (transient), `retryable=False` for `result.error` failures (validation/quality gate — permanent until data changes).

4. After the loop, build the `BatchIngestResponse` with accepted_ids and rejected.

5. Return HTTP 207 when there are BOTH accepted and rejected items (partial success). Return 200 when all succeed. Use `JSONResponse` with explicit `status_code` for the 207 case. Import `JSONResponse` from `fastapi.responses`.

Do NOT change the endpoint path or request schema — only the response tracking and status code logic.
  </action>
