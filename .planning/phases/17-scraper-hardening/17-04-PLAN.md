<objective>
Harden the scraper sync pipeline: selective hash updates based on backend acceptance, soft-delete for removed products, and archive notification to backend.

Purpose: Currently the scraper updates SQLite hashes for ALL products regardless of backend acceptance, meaning failed products are never retried. And `mark_removed` hard-deletes from SQLite, losing tracking state. These two bugs silently drop products.
Output: BackendSync returns per-item accepted_ids, pipeline only updates hashes for accepted products, mark_removed uses soft-delete, removed products trigger backend archival.
</objective>

1. Add a `SyncResult` dataclass to schemas.py (or as a simple class in backend_sync.py):
   ```python
   @dataclass
   class SyncResult:
       created: int = 0
       updated: int = 0
       failed: int = 0
       accepted_ids: list[str] = field(default_factory=list)
   ```

2. Change `push_products` return type from `int` to `SyncResult`. Accumulate `accepted_ids` from the response JSON across all chunks:
   - Parse `data.get("accepted_ids", [])` from each batch response
   - Extend `result.accepted_ids` with those IDs
   - Track created/failed counts as before
   - Handle both HTTP 200 and 207 as success (both return the same JSON body)

3. Change `update_products` return type from `int` to `SyncResult` with the same accepted_ids tracking.

4. Add `archive_products` method to BackendSync:
   ```python
   async def archive_products(self, store_id: str, external_ids: list[str]) -> int:
   ```
   - POST to `{self.api_url}/api/products/archive/batch` with `{"store_id": store_id, "external_ids": external_ids}`
   - Return `archived_count` from response
   - Log the count
   - If BACKEND_API_URL not set (sync disabled), return 0

Do NOT change the payload format sent TO the backend (ProductCreate format is unchanged). Only change what we READ from the response.
  </action>

  <name>Task 2: Selective hash update + soft-delete mark_removed + archive call in pipeline</name>
  <files>apps/scraper/scraper/pipeline.py, apps/scraper/scraper/change_detection.py</files>
  <action>
1. In `change_detection.py`, change `mark_removed` from DELETE to soft-delete:
   - Instead of `DELETE FROM product_hashes WHERE ...`, UPDATE to set `removed_at = datetime('now')`.
   - First, update the `initialize` method to add `removed_at TEXT` column to the CREATE TABLE IF NOT EXISTS statement. For existing databases, add an ALTER TABLE migration: check if column exists, add if not.
   - Update `detect_changes` to exclude rows where `removed_at IS NOT NULL` when building the `existing` dict — removed products should not appear as "existing" (they were already processed as removed).
   - Add a method to clear `removed_at` if a product re-appears: when a product is in the `new` list but has a row with `removed_at IS NOT NULL`, treat it as returning (clear removed_at, treat as changed instead of new).

2. In `pipeline.py`, update `run_store`:
   - After `sync.push_products(report.new)`, get the `SyncResult` back.
   - Only update hashes for products whose `external_id` is in `result.accepted_ids`:
     ```python
     result = await sync.push_products(report.new)
     accepted_new = [p for p in report.new if p.external_id in set(result.accepted_ids)]
     if accepted_new:
         await detector.update_hashes(store.id, accepted_new)
     ```
   - Same pattern for `sync.update_products(report.changed)`.
   - Replace the current unconditional `await detector.update_hashes(store.id, all_synced)` with the selective approach above.
   - After `detector.mark_removed(store.id, report.removed_ids)`, also call `await sync.archive_products(store.id, report.removed_ids)` to notify the backend.

Do NOT change the scraping or change detection logic — only the post-sync hash update flow and mark_removed behavior.
  </action>

  <done>Pipeline only updates hashes for backend-accepted products. Rejected products keep old/no hash and are retried on next run. mark_removed soft-deletes instead of hard-deleting. Removed products trigger backend archival.</done>
