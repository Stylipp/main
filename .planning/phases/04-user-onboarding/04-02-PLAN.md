- Qdrant is single source of truth for vectors (from 02-01, 03-01)

- Cosine similarity with 768-dim vectors (from 02-01)
- asyncio.to_thread for CPU-intensive ops (from 03-01)

**Research findings (from 04-RESEARCH.md):**
- Modified Rocchio formula: 0.3 * photo_embedding + 1.0 * avg(liked) - 0.25 * avg(disliked)
- PROJECT.md specifies gamma=0.7 but research recommends 0.25 (literature standard range 0.15-0.25)
- Price profile: IQR-based from liked items only (not dislikes)
- Quality gate validates before embedding (existing service)

  <name>Task 2: Create calibration completion endpoint with user vector and price profile</name>
  <files>apps/backend/src/features/onboarding/service/user_vector.py, apps/backend/src/features/onboarding/service/service.py, apps/backend/src/features/onboarding/schemas/schemas.py, apps/backend/src/features/onboarding/router/router.py, apps/backend/src/core/qdrant.py, apps/backend/src/main.py</files>
  <action>
1. In service/user_vector.py, create pure functions:
   - `compute_user_vector(photo_embeddings: list[list[float]], liked_embeddings: list[list[float]], disliked_embeddings: list[list[float]]) -> list[float]`:
     - Use modified Rocchio formula: `0.3 * avg(photo_embeddings) + 1.0 * avg(liked_embeddings) - 0.25 * avg(disliked_embeddings)`
     - L2-normalize the result for cosine similarity search
     - Return as list[float] (768-dim)
     - NOTE: gamma=0.25 deviates from PROJECT.md's 0.7 per research recommendation (04-RESEARCH.md). The 0.7 weight is too aggressive for sparse calibration data (only ~5-6 dislikes). Standard Rocchio range is 0.15-0.25.
     - Use numpy for vector operations. Import at function level or module level.

   - `initialize_price_profile(liked_prices: list[float]) -> dict`:
     - If fewer than 3 prices: return wide defaults (median if available, else 50, range 0.3x-3.0x)
     - Otherwise: compute median, Q1, Q3, IQR. Return {price_min: max(0, Q1-1.5*IQR), price_max: Q3+1.5*IQR, price_median: median, price_std: std}
     - Only use liked items (not disliked — user may dislike style, not price)

2. In schemas/schemas.py, add:
   - `CalibrationCompleteRequest(BaseModel)`: photo_embeddings (list[list[float]]), liked_product_ids (list[str]), disliked_product_ids (list[str])
   - `CalibrationCompleteResponse(BaseModel)`: success (bool), onboarding_completed (bool), cluster_count (int), price_profile (dict | None)

3. In service/service.py, add to OnboardingService:
   - `async def complete_calibration(self, user_id: UUID, request: CalibrationCompleteRequest, session: AsyncSession) -> CalibrationCompleteResponse`:
     - Validate: at least 1 photo embedding, at least 1 liked product
     - Fetch liked product embeddings from Qdrant "products" collection by product IDs (use qdrant_client.retrieve with IDs)
     - Fetch disliked product embeddings similarly
     - Call compute_user_vector() with photo embeddings + liked/disliked embeddings
     - Call initialize_price_profile() with liked product prices (query from PostgreSQL products table)
     - Store user vector in Qdrant "user_profiles" collection (upsert point with user_id as point ID and payload)
     - Update User in PostgreSQL: set onboarding_completed=True, price_profile=computed profile, display_name if provided
     - Return CalibrationCompleteResponse

4. In core/qdrant.py:
   - Add `ensure_user_profiles_collection()` alongside existing ensure_cluster_collection()
   - Config: name from settings (add `user_profiles_collection: str = "user_profiles"` to Settings), 768-dim, cosine distance
   - Idempotent creation (same pattern as cluster collection)

5. In main.py:
   - Import and register onboarding router: `app.include_router(onboarding_router, prefix="/api")`
   - Call `await ensure_user_profiles_collection()` in lifespan alongside ensure_cluster_collection()

**Avoid:** Don't store user vector in PostgreSQL (Qdrant is source of truth for vectors). Don't use disliked items for price profiling. Don't block the event loop — use asyncio.to_thread for numpy operations if they become heavy (for 768-dim vectors with <20 items, this is fast enough synchronously). Don't hardcode collection names — use settings.
