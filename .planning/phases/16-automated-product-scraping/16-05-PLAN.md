  <name>Task 1: Create vector sync service for embedding new and changed products into Qdrant</name>
  <files>apps/backend/src/features/scraper/service/vector_sync_service.py</files>
  <action>
    Create VectorSyncService that handles the Qdrant side of product sync.

    Class **VectorSyncService**:

    - `__init__(self, embedding_service: EmbeddingService, quality_gate: QualityGateService, qdrant_client: AsyncQdrantClient, collection_name: str = "products")`:
      Store all dependencies. Reuse the SAME embedding and quality gate services as the main ingestion pipeline — do NOT create new instances.

    - `async def sync_products_to_qdrant(self, products: list[Product]) -> tuple[int, int]`:
      Process a list of Product model instances (from PostgreSQL) into Qdrant:
      1. For each product:
         a. Fetch image from product.image_url using httpx.AsyncClient (same pattern as IngestionService._fetch_image)
         b. Validate via quality_gate.validate_image(image, file_size)
         c. If quality fails: log warning, skip this product (don't block others)
         d. Generate embedding via embedding_service.get_embedding(image)
         e. Upsert to Qdrant: PointStruct with id=str(product.id), vector=embedding, payload={product_id, store_id, price, created_at}
      2. Process sequentially (embedding is CPU-bound with semaphore limiting)
      3. Log progress every 25 products: "{processed}/{total} products embedded"
      4. Return (success_count, failed_count)

    - `async def remove_from_qdrant(self, product_ids: list[str])`:
      Remove vectors for products that have been deleted:
      Use qdrant_client.delete(collection_name, points_selector=PointIdsList(points=product_ids))
      Log: "{n} vectors removed from Qdrant"
      Note: For Phase 16, we're NOT deleting products from PostgreSQL (they may have user interactions), but if we ever do, this method handles the Qdrant cleanup.

    - `async def should_recluster(self, changed_count: int, total_count: int, threshold: float) -> bool`:
      Simple threshold check: return changed_count / max(total_count, 1) > threshold
      Log the decision: "Re-cluster check: {changed_count}/{total_count} = {ratio:.1%} (threshold: {threshold:.1%}) → {yes/no}"

    - `async def trigger_recluster(self, session: AsyncSession)`:
      Wrap ClusteringService.rebuild_clusters(session) call:
      1. Import and instantiate ClusteringService with qdrant_client and settings
      2. Call rebuild_clusters(session)
      3. Log: "Re-clustering complete"
      Use asyncio.to_thread if needed (follows Phase 03-01 pattern for CPU-intensive sklearn ops).

    Import types: Product from models, EmbeddingService, QualityGateService, AsyncQdrantClient, PointStruct, PointIdsList from qdrant_client.models, ClusteringService, AsyncSession.

    Important: Reuse the exact same Qdrant payload structure as IngestionService (product_id, store_id, price, created_at) to maintain consistency.

  <action>
    Add a more efficient batch processing method to VectorSyncService that processes images in batches rather than one-by-one:

    - `async def sync_products_batch(self, products: list[Product], batch_size: int = 8) -> tuple[int, int]`:
      Optimized version of sync_products_to_qdrant:
      1. Pre-fetch all images concurrently (up to 4 concurrent fetches via asyncio.Semaphore):
         - Create semaphore = asyncio.Semaphore(4)
         - For each product, create a task that fetches image within semaphore
         - Gather all fetch tasks
         - Collect: list of (product, image, file_size) tuples, logging failures
      2. Validate all images through quality gate (this is CPU-cheap, no batching needed)
      3. Generate embeddings in batches using embedding_service.get_embeddings_batch(images, batch_size=batch_size)
         - The existing EmbeddingService.get_embeddings_batch already handles batched inference
      4. Bulk upsert to Qdrant: collect all PointStruct objects and upsert in a single call
         - qdrant_client.upsert(collection_name, points=[all_points])
      5. Return (success_count, failed_count)

    Make sync_products_to_qdrant call sync_products_batch internally (keep the simple interface, use batch implementation).

    This is important because nightly runs may process 50-200 changed products — batching image fetches and embeddings makes this 3-5x faster than sequential.

    Error handling: If any individual image fetch or embedding fails, skip that product but continue with the rest. Log each failure.
